# 模型先知发现机制改造方案（Roadmap）

## 1. 背景与目标

当前模型先知已经具备基础的“近时发现”能力，但仍存在以下问题：

- 新鲜度判断依赖 `extra_data.updated_at`（JSON 字段），查询和索引能力受限。
- “发现（discovery）”与“展示（list models）”的时间窗口策略还未完全统一。
- 缺乏可观测指标（误报率/召回率）来验证先知质量。

本方案目标：

1. **只让近期活跃仓库进入主视图**（默认 7 天，可扩展）。
2. **统一源头过滤 + 入库过滤 + 展示过滤策略**，减少语义偏差。
3. **引入可量化质量指标**，让策略优化有数据支撑。

---

## 2. 当前实现（As-Is）

- `GET /exploration/models` 已默认按 `max_update_age_days=7` 过滤。
- 列表过滤信号已切换为结构化字段 `last_activity_at`（索引字段），并保留迁移期兼容策略用于报告刷新判定。
- 发现侧（例如 GitHub）已在检索阶段使用 `pushed:>cutoff` 及二次过滤。

这说明方向正确，但还没有达到“结构化可扩展”的最优状态。

---

## 3. 目标架构（To-Be）

### 3.1 数据模型增强（核心）

新增结构化字段（建议在 `discovered_models` 表）：

- `last_activity_at`：最近一次有效活动时间（可索引）
- `activity_type`：活动类型（`new_release_tag` / `weights_update` / `doc_update` / ...）
- `activity_confidence`：活动可信度（0-100）

收益：

- 避免 JSON 提取 + 时间转换函数带来的查询复杂度。
- 能在数据库层高效分页、排序与筛选。

### 3.2 统一策略

1. **源头过滤**：平台 API 先按时间窗口拉取，降低噪音。
2. **入库过滤**：低可信更新进入观察池，不进入主候选池。
3. **展示过滤**：默认 7 天，支持 14/30 天切换用于回溯。

### 3.3 排序逻辑升级

建议使用多因子评分：

`rank = freshness * α + quality * β + impact * γ + watch_hit * δ`

避免“仅新不强”或“很强但过旧”的单维偏差。

## 4. 分阶段开发计划

## Phase 1（当前迭代）

- [x] 文档化改造路线。
- [x] 固化默认“近 7 天新鲜度”过滤并提升时间比较稳定性。
- [x] 把新鲜度过滤抽象成可复用函数，便于后续复用到统计/报表接口。

**验收标准**：

- 默认列表不再出现长期未更新项目。
- 时间过滤在混合时区格式下保持稳定。

## Phase 2

- [x] 增加结构化字段 `last_activity_at/activity_type/activity_confidence`。
- [x] 数据迁移与回填脚本（从 `extra_data` 迁移）。
- [x] 列表查询改用结构化字段，并建立索引。

**验收标准**：

- 列表查询性能稳定。
- 新旧数据兼容，线上无中断。

## Phase 3

- [x] 引入“观察池”概念。
- [x] 加入先知质量指标（误报率/召回率）看板。
- [x] 基于指标迭代阈值与权重。

**验收标准**：

- 误报率可持续下降，召回率不显著下降。

---

## 5. 风险与回滚

### 风险

- 新增字段/迁移可能带来短期兼容成本。
- 活动类型规则不当会影响先知准确度。

### 回滚策略

- 查询层保留旧信号兜底（`extra_data.updated_at` + `release_date`）。
- 特性开关控制新排序策略，异常时快速回退。

---

## 6. 下一步（建议）

下一步建议推进 **Phase 4（运营化）**：补齐前端质量看板交互、阈值配置化、以及按组织/来源维度的质量趋势分析。


## 7. 当前落地说明（新增）

- 列表接口默认使用 `last_activity_at` 进行新鲜度筛选，并可通过 `status` 过滤观察池模型。
- 统计接口补充 `watching_models`、`active_models_7d`、`false_positive_rate`、`recall_rate_proxy` 指标。
- 统计接口支持 `stale_days` 与 `recall_window_days` 参数，便于按指标口径快速调参。
