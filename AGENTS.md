# Agent 基础知识

## Windows 下的 UTF-8 文件处理

在 Windows，尤其是 PowerShell 环境下，不能假设文本文件会被自动按 UTF-8 正确读取或写回。默认编码、本地代码页、以及 BOM 与无 BOM 的差异，都可能导致 UTF-8 文件被误判或被写坏。

常见风险：

- 读取时把 UTF-8 文本按系统代码页解释，出现乱码。
- 写入时把 UTF-8 文件误写成 ANSI、UTF-16LE 或其他编码，直接破坏文件。
- 文件表面上看似只改了少量内容，但实际整个文件的编码已经被改变。

## 必须遵守的规则

1. 在 Windows 下读取文本文件时，不能依赖默认编码；必须显式按 UTF-8 读取，必要时先确认原始编码。
2. 在 Windows 下写入、覆盖或生成文本文件时，不能依赖默认编码；必须显式按 UTF-8 写回，并避免无意改变 BOM 状态。
3. 编辑包含中文或其他非 ASCII 字符的文件前，先确认文件内容显示正常，再进行修改。
4. 如果需要通过 shell 写文件，必须显式指定编码，不能直接依赖默认 `Set-Content`、`Out-File` 等行为。
5. 如果对文件编码没有把握，先停止写入，先检查，再修改；不能在编码未确认的情况下冒险覆盖文件。
6. 修改完成后，要再次检查关键非 ASCII 内容是否仍然正常，确认没有出现乱码、替换字符或编码异常。

## 执行提醒

- “能读出来”不等于“编码正确”。
- “只改一小处”也可能把整个 UTF-8 文件写坏。
- Windows 下处理 UTF-8 文件时，编码安全优先于操作速度。
